# CIFAR-10-C em Federated Learning

> Revisão de literatura — 20/05/2026
>
> Contexto: como papers de FL usam CIFAR-10-C, e como adaptar o protocolo
> para o experimento de drift temporal com sync vs. async.

## 1. O Dataset

CIFAR-10-C (Hendrycks & Dietterich, ICLR 2019) é o benchmark canônico de
robustez a corrupções. São **19 tipos de corrupção** aplicadas sobre as 10.000
imagens de teste do CIFAR-10, em **5 níveis de severidade** (1 a 5):

| Grupo | Corrupções |
|---|---|
| Noise | Gaussian noise, shot noise, impulse noise, speckle noise |
| Blur | defocus blur, glass blur, motion blur, zoom blur, Gaussian blur |
| Weather | snow, frost, fog, spatter |
| Digital | brightness, contrast, elastic transform, pixelate, JPEG compression, saturate |

**Vantagens sobre manipular imagens manualmente:**
- É pré-computado e canônico — todo mundo usa os mesmos arquivos
- Tem carga imediata via `torchvision.datasets.CIFAR10` com `transform`
- Cada corrupção é um domínio visual distinto, simulando exatamente o que o
  professor descreveu (ex: "brightness" = verão claro, "fog" = neblina de
  inverno)
- As 10 classes estão sempre presentes em todas as corrupções → sem
  catastrophic forgetting

## 2. Como CIFAR-10-C é Usado em FL

### 2.1 Protocolo Principal: Federated Domain Generalization (FDG)

O uso mais comum: cada cliente recebe **uma corrupção diferente** como seu
"domínio" local. O modelo global é testado em uma corrupção que **nenhum
cliente viu** durante o treinamento. O objetivo é generalização fora da
distribuição.

**Setup típico (ex: PerAda, DiPrompT, FedDG-GA):**

```
Cliente 1: CIFAR-10-C (Gaussian noise, severity 3)
Cliente 2: CIFAR-10-C (brightness, severity 3)
Cliente 3: CIFAR-10-C (contrast, severity 3)
...
Cliente 15: CIFAR-10-C (pixelate, severity 3)

Teste: CIFAR-10-C (fog, severity 3)  ← corrupção NÃO usada por nenhum cliente
```

Em alguns trabalhos, o CIFAR-10 limpo (sem corrupção) é usado também como um
"domínio" adicional.

### 2.2 Papers Principais que Usam CIFAR-10-C

**PerAda** — Xie et al. (CVPR 2024)
- Paradigma: Personalized FL com adapters
- Resultado no CIFAR-10-C: +5.23% de generalização OOD vs. baselines
- Modelo: ResNet-18 pré-treinado + adapters (~12.6% dos parâmetros)
- Setup: cada cliente tem um domínio diferente; teste em domínio não visto

**DiPrompT** — Bai et al. (CVPR 2024)
- Paradigma: Federated Domain Generalization sem labels de domínio
- Usa CIFAR-10-C e PACS como benchmarks
- Não requer que o número de domínios = número de clientes
- Modelo: CLIP pré-treinado + prompt tuning

**FedUV** — Son et al. (CVPR 2024)
- Paradigma: FL com regularização de uniformidade e variância
- Testa **tanto label-shift quanto feature-shift**
- CIFAR-10-C é usado para o cenário de **feature-shift**
- Modelo: ResNet-18

**FedDG-GA** — Zhang et al. (CVPR 2023)
- Paradigma: Domain generalization via ajuste de agregação no servidor
- CIFAR-10-C e PACS como benchmarks
- Métrica: accuracy no domínio alvo não visto

**FedFA** — ECCV 2024
- Paradigma: Feature diversification and adaptation
- Usa CIFAR-10-C para avaliar generalização entre domínios visuais distintos

**FedHEAL** — Chen et al. (CVPR 2024)
- Paradigma: Fair FL under domain skew
- CIFAR-10-C usado para criar desbalanceamento de domínio entre clientes

### 2.3 Modelos Utilizados

A literatura de FDG com CIFAR-10-C converge em:
- **ResNet-18** (mais comum, usado por PerAda pré-ViT, FedUV, FedDG-GA)
- **ResNet-50** (alguns trabalhos maiores)
- **CLIP/ViT pré-treinado + prompt tuning** (tendência em 2024: DiPrompT, FedOTP)
- CNNs pequenas (LeNet, CNN-4) são **inadequadas** para CIFAR-10-C — a
  performance absoluta é muito baixa e a variância domina o sinal

### 2.4 Métricas Utilizadas

- **Target domain accuracy**: acurácia no domínio de teste não visto (a métrica
  principal de FDG)
- **Average accuracy across all target domains**: média sobre múltiplos
  domínios alvo
- **Worst-case accuracy**: acurácia no pior domínio alvo (robustez)
- **Generalization gap**: diferença entre acurácia no domínio fonte e no alvo

## 3. Adaptação para Drift Temporal

O uso em FDG é *cross-client espacial* (cada cliente = 1 domínio fixo). Para
drift temporal, precisamos de um protocolo diferente:

### Protocolo Proposto: CIFAR-10-C Sazonal

```
Fase 1 (0 → T_drift): TODOS os clientes usam corrupção "brightness" (severidade 3)
                       → Modelo aprende em domínio claro

Fase 2 (T_drift → 2×T_drift): TODOS os clientes usam corrupção "fog" (severidade 3)
                               → Modelo precisa adaptar a domínio com neblina

Fase 3 (2×T_drift → 3×T_drift): TODOS os clientes usam corrupção "contrast" (severidade 3)
                                 → Mais uma transição
```

**Por que todos os clientes com a mesma corrupção por fase?**
Porque o drift é *temporal*, não espacial. O cenário do professor é: as
câmeras estão todas na mesma cidade, passando pela mesma mudança climática ao
mesmo tempo. A heterogeneidade entre clientes continua vindo dos tiers de
velocidade (fast/medium/slow), não de domínios visuais diferentes.

### Complexidade de Implementação

**O que muda no código:**
- `data_loader.py`: adicionar suporte a carregar CIFAR-10-C com corrupção
  específica (via `torchvision.datasets.CIFAR10` ou `CIFAR10C` package)
- `data_split.py`: a partição IID continua igual — a corrupção é uma camada
  acima, aplicada na fase atual
- `temporal_drift.py`: no momento do drift, recarrega o dataset com a nova
  corrupção e redistribui entre clientes

**O que NÃO muda:**
- `client.py`, `server.py`, `models.py`, `plot_accuracy.py`
- O formato de saída JSON (`{loss, accuracy, time}`)
- Os parâmetros de linha de comando existentes

### Métricas Adicionais para Este Experimento

Além das métricas padrão (max_acc, time_to_acc, etc.):

- **Recovery time após drift**: tempo virtual para voltar a X% da acurácia
  pré-drift
- **Sustained accuracy**: média da acurácia ponderada pelo tempo (penaliza
  vales longos)
- **Per-corruption accuracy**: acurácia quebrada por fase/corrupção

### Hipóteses Testáveis

1. **H1**: Async se recupera mais rápido que sync após cada drift (menor
   recovery time) porque converge mais rápido dentro da nova fase

2. **H2**: Sync atinge maior acurácia de pico dentro de cada fase *se a fase
   durar o suficiente* (T_drift > crossover sync/async)

3. **H3**: Com T_drift curto (< crossover), async tem maior *sustained
   accuracy* porque sync nunca atinge o platô antes do próximo drift

4. **H4**: O crossover sync/async se desloca para a direita com drift vs. sem
   drift (ou seja, o sync precisa de *mais* tempo para valer a pena quando há
   mudanças de domínio)

## 4. Observação sobre o Modelo

A literatura de FDG com CIFAR-10-C usa ResNet-18 ou ViT. O `CNNCIFAR10`
atual (4 conv layers, ~1M params) pode ter performance absoluta baixa em
CIFAR-10-C (~40-50% de teto em corrupções severidade 3), o que comprimiria a
diferença entre sync e async.

**Sugestão**: considerar subir para ResNet-18 (pré-treinada ou não) para o
experimento com CIFAR-10-C. O `torchvision.models.resnet18` já existe e exigiria
só adaptar o `get_model()` para incluir a opção.
